# TP 4 — Profiler, corriger une fuite mémoire et scaler en cluster

> Module 4 / Gestion de la performance — Type : TP guidé — Difficulté 4/5 — Durée ~1h30

## Mise en situation

Vous reprenez une API Express de catalogue produits qui montre deux symptômes en production : sa consommation mémoire (RSS) monte continuellement jusqu'à redémarrage forcé, et une route de calcul de remise (CPU-bound) sature un seul cœur tandis que la machine en possède huit. Votre mission : instrumenter, prouver la fuite, la corriger, puis scaler le calcul lourd sur tous les cœurs et mesurer le gain de débit.

## Objectifs

- Profiler une API Express avec `--inspect` et clinic.
- Identifier une fuite mémoire par comparaison de heap snapshots, puis la corriger.
- Scaler un endpoint CPU-bound avec le module `cluster`.
- Mesurer les débits avant/après avec `autocannon`.

## Prérequis techniques

```bash
node --version          # v20.x ou v22.x attendu
npm init -y
npm install express lru-cache
npm install -g autocannon clinic
# Chrome/Chromium installé pour ouvrir chrome://inspect (heap snapshots)
```

## Architecture cible

```text

   autocannon  ───▶   │   Process PRIMARY (cluster) │

        └───────────┘         └───────────┘           └───────────┘
```

## Étapes

### Étape 1 — Reproduire et profiler la fuite mémoire (durée ~30 min)

**Objectif :** disposer d'une app fuyante, prouver la fuite par snapshots et localiser l'objet retenu.

Créez `app.js` :

```js
const express = require('express');
const app = express();

// FUITE : cache global non borné, indexé par requête
const cacheVues = {};

app.get('/produit/:id', (req, res) => {
  const cle = `${req.params.id}-${Date.now()}-${Math.random()}`;
  cacheVues[cle] = {
    id: req.params.id,
    vu: new Date().toISOString(),
    blob: 'x'.repeat(10_000), // ~10 Ko par entrée
  };
  res.json({ id: req.params.id, entrees: Object.keys(cacheVues).length });
});

app.get('/health', (req, res) => res.json({ status: 'ok' }));

module.exports = app;
if (require.main === module) {
  app.listen(3000, () => console.log('App sur http://localhost:3000'));
}
```

Lancez avec inspection puis générez de la charge :

```bash
node --inspect app.js
# Dans un autre terminal :
autocannon -c 50 -d 20 "http://localhost:3000/produit/42"
```

Ouvrez `chrome://inspect`, cliquez « inspect », onglet **Memory**. Prenez un heap snapshot, relancez `autocannon`, prenez un second snapshot, puis comparez en mode « Comparison ».

**Point de contrôle :** vous identifiez l'objet `cacheVues` (ou ses entrées) comme principal contributeur à la croissance du heap entre les deux snapshots, et `process.memoryUsage().heapUsed` ne redescend pas après un `global.gc()` (lancé via `node --expose-gc`).

### Étape 2 — Corriger la fuite (durée ~25 min)

**Objectif :** borner la mémoire pour que le heap se stabilise sous charge soutenue.

Remplacez le cache non borné par un cache LRU borné :

```js
const { LRUCache } = require('lru-cache');

const cacheVues = new LRUCache({
  max: 500,                  // 500 entrées maximum
  ttl: 1000 * 60,            // expiration au bout d'une minute
});

app.get('/produit/:id', (req, res) => {
  const cle = req.params.id;                 // clé stable, pas de Date/random
  let entree = cacheVues.get(cle);
  if (!entree) {
    entree = { id: cle, vu: new Date().toISOString(), blob: 'x'.repeat(10_000) };
    cacheVues.set(cle, entree);
  }
  res.json({ id: cle, entrees: cacheVues.size });
});
```

Relancez la même charge et reprenez deux snapshots espacés.

**Point de contrôle :** sous charge identique, `cacheVues.size` plafonne (≤ 500) et `heapUsed` se stabilise dans une fourchette au lieu de croître linéairement. La comparaison de snapshots ne montre plus de croissance illimitée des entrées du cache.

### Étape 3 — Scaler le calcul CPU-bound en cluster (durée ~35 min)

**Objectif :** exploiter tous les cœurs pour un endpoint CPU-bound et mesurer le gain de débit.

Ajoutez à `app.js` une route lourde :

```js
const crypto = require('node:crypto');
app.get('/remise/:montant', (req, res) => {
  let h = req.params.montant;
  for (let i = 0; i < 2_000_000; i++) {
    h = crypto.createHash('sha256').update(h).digest('hex');
  }
  res.json({ montant: req.params.montant, signature: h.slice(0, 16), pid: process.pid });
});
```

Mesurez d'abord en mono-process :

```bash
node app.js
autocannon -c 100 -d 20 "http://localhost:3000/remise/1000"   # noter req/s et p99
```

Créez `cluster.js` :

```js
const cluster = require('node:cluster');
const os = require('node:os');

if (cluster.isPrimary) {
  const n = os.availableParallelism();
  console.log(`Primary ${process.pid} démarre ${n} workers`);
  for (let i = 0; i < n; i++) cluster.fork();
  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} mort, relance`);
    cluster.fork();
  });
} else {
  const app = require('./app');
  app.listen(3000, () => console.log(`Worker ${process.pid} en écoute`));
}
```

Relancez en cluster et remesurez :

```bash
node cluster.js
autocannon -c 100 -d 20 "http://localhost:3000/remise/1000"
```

**Point de contrôle :** sous cluster, les réponses font apparaître plusieurs `pid` différents (répartition entre workers) et le débit (req/s) sur `/remise` augmente sensiblement par rapport au mono-process, sans effondrer la latence des autres routes.

## Livrable attendu

- `app.js` corrigé (cache LRU borné) avec route `/remise`.
- `cluster.js` lançant N workers.
- Un court relevé de mesures `autocannon` : débit et p99 sur `/remise` en mono-process puis en cluster, et la stabilisation du heap avant/après correction de la fuite.

## Dépannage

<details>
<summary>chrome://inspect ne montre pas ma cible</summary>

Vérifiez que le process tourne bien avec `--inspect` et que le port 9229 n'est pas déjà pris. Cliquez sur « Open dedicated DevTools for Node », ou ajoutez la cible via « Configure » (`localhost:9229`).
</details>

<details>
<summary>global.gc is not a function</summary>

Lancez Node avec `node --expose-gc --inspect app.js`. Sans ce flag, `global.gc()` n'est pas disponible.
</details>

<details>
<summary>autocannon : Address already in use (EADDRINUSE)</summary>

Un process écoute déjà sur 3000. Tuez-le (`lsof -i :3000` puis `kill`) avant de relancer. En cluster, ne lancez pas aussi `app.js` en parallèle.
</details>

<details>
<summary>Le débit en cluster n'augmente pas</summary>

Vérifiez que vous mesurez bien la route CPU-bound (`/remise`) et non une route I/O triviale. Confirmez que plusieurs workers ont été forkés (logs « Worker ... en écoute ») et que la machine a plusieurs cœurs (`os.availableParallelism()`).
</details>

## Pour aller plus loin

- Remplacez le calcul `/remise` par un `worker_threads` avec pool, et comparez au cluster en termes de mémoire consommée.
- Ajoutez `perf_hooks.monitorEventLoopDelay()` et exposez le lag de l'event loop par worker.
- Mettez l'app derrière un reverse proxy (nginx) et comparez la répartition de charge avec celle native du module `cluster`.
