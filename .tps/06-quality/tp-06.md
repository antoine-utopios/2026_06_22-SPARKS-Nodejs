# TP 6 — Outiller une API Express : debug, profiling, lint et erreurs

> Module 6 / Contrôle de qualité — TP guidés — Difficulté 4/5 — Durée ~1h30

## Mise en situation

Votre équipe hérite d'une API Express de catalogue produits. Elle est lente sur une route, plante parfois sans laisser de trace exploitable, et personne n'a configuré de linter. Avant de la passer en production, on vous demande de l'outiller : pouvoir la debugger pas à pas, identifier la route lente par profiling, appliquer un lint, et rendre la gestion d'erreurs robuste (classes, middleware, handlers de process).

## Objectifs

- Configurer le debug `--inspect` et une configuration VS Code utilisable au pas-à-pas
- Profiler l'application pour identifier la route lente
- Mettre en place ESLint avec des règles de qualité et de complexité
- Industrialiser la gestion d'erreurs : classes, middleware centralisé, handlers `uncaughtException` / `unhandledRejection`, et sortie propre

## Prérequis techniques


## Étapes

### Étape 1 — Debug pas à pas (≈25 min)

Objectif : lancer l'API en mode inspection et poser un breakpoint dans une route depuis VS Code.

Créez `src/routes/products.js` et `src/server.js` avec au moins une route `GET /products/:id`, puis la configuration de debug.

```bash
# Lancement en mode inspection, suspendu au démarrage
node --inspect-brk src/server.js
```

`.vscode/launch.json` :

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "node",
      "request": "launch",
      "name": "Debug API",
      "program": "${workspaceFolder}/src/server.js",
      "runtimeArgs": ["--inspect"],
      "skipFiles": ["<node_internals>/**"],
      "console": "integratedTerminal"
    }
  ]
}
```

Posez un breakpoint dans le handler `GET /products/:id`, lancez la configuration, appelez la route et inspectez `req.params` via la vue Watch.

Point de contrôle : l'exécution s'arrête sur votre breakpoint et vous voyez la valeur de `req.params.id` dans la fenêtre de variables.

### Étape 2 — Profiling de la route lente (≈30 min)

Objectif : ajouter une route volontairement coûteuse, puis l'identifier par profiling.

Ajoutez une route `GET /products/report` qui effectue un calcul lourd (boucle synchrone). Générez de la charge et profilez.

```bash
# Option A : log brut V8 puis rapport
node --prof src/server.js &
npx autocannon -d 8 http://localhost:3000/products/report
# arrêter le process, puis :
node --prof-process isolate-0x*.log > processed.txt

# Option B : clinic
npx clinic doctor -- node src/server.js
# (générer la charge, puis Ctrl+C)
npx clinic flame -- node src/server.js
```

Point de contrôle : dans `processed.txt` (section Summary / ticks) ou dans le flamegraph clinic, la fonction de calcul de `/products/report` apparaît comme la plus consommatrice de CPU.

### Étape 3 — Lint et gestion d'erreurs robuste (≈35 min)

Objectif : configurer ESLint puis centraliser les erreurs et sécuriser le process.

```bash
npx eslint .          # doit remonter les problèmes initiaux
```

`eslint.config.js` :

```js
import js from "@eslint/js";

export default [
  js.configs.recommended,
  {
    languageOptions: { ecmaVersion: 2023, sourceType: "module" },
    rules: {
      "no-unused-vars": "error",
      "eqeqeq": "error",
      "complexity": ["warn", 10],
      "max-depth": ["warn", 4]
    }
  }
];
```

Implémentez ensuite :

- `src/errors.js` : `AppError`, `NotFoundError` (404), `ValidationError` (400)
- `src/middleware/errorHandler.js` : middleware `(err, req, res, next)` centralisé
- dans `src/server.js` : enregistrement du middleware après les routes, plus les handlers de process

```js
process.on("uncaughtException", (err) => {
  console.error("uncaughtException", err);
  process.exit(1);
});
process.on("unhandledRejection", (reason) => {
  console.error("unhandledRejection", reason);
  process.exit(1);
});

const server = app.listen(3000);
process.on("SIGTERM", () => server.close(() => process.exit(0)));
```

Faites en sorte que `GET /products/abc` renvoie 400, `GET /products/999` renvoie 404, et qu'une erreur inattendue renvoie 500 sans fuiter de détails.

Point de contrôle : `npx eslint .` ne remonte plus d'erreur bloquante, les codes HTTP 400/404/500 sont corrects, et un `SIGTERM` ferme proprement le serveur.

## Livrable attendu

Un projet `tp06-quality` fonctionnel comprenant : la configuration de debug VS Code, un fichier `processed.txt` (ou un rapport clinic) pointant la route lente, une configuration ESLint sans erreur bloquante, et une gestion d'erreurs centralisée avec handlers de process et arrêt propre.

## Dépannage

<details>
<summary>« Cannot connect to runtime process » dans VS Code</summary>

Le port 9229 est peut-être déjà occupé par une autre instance. Fermez les process Node en cours (`lsof -i:9229`), ou changez le port avec `--inspect=9230` et alignez `"port"` dans la config attach.

</details>

<details>
<summary>Aucun fichier `isolate-*.log` généré</summary>

Le log n'est écrit qu'à l'arrêt propre du process. Assurez-vous d'avoir lancé `node --prof` (et non via un wrapper) et d'arrêter le process après avoir généré la charge.

</details>

<details>
<summary>ESLint ne trouve pas de configuration</summary>

Depuis ESLint 9, la flat config `eslint.config.js` est utilisée par défaut. Vérifiez que le fichier est à la racine et que `package.json` a `"type": "module"`.

</details>

<details>
<summary>Les rejets async ne déclenchent pas le middleware (Express 4)</summary>

En Express 4, enveloppez les handlers : `const asyncHandler = (fn) => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next);`. Express 5 le fait nativement.

</details>

- Mesurez l'amélioration après avoir rendu la route `/products/report` asynchrone ou paginée
- Intégrez `eslint` dans un hook de pré-commit (lint-staged + husky)
