# TP 7 — Service de calcul signé exposé en flux

> Module 7 / Modules avancés — TP guidé. Difficulté 4/5, durée estimée 1h30.

## Mise en situation

Votre équipe doit exposer un endpoint qui réalise un calcul coûteux (par exemple une dérivation de clé ou une agrégation lourde), garantit l'intégrité du résultat par une signature cryptographique, et renvoie le tout en flux pour ne pas bloquer le serveur ni saturer la mémoire. Vous allez combiner trois modules avancés de Node.js dans une seule route Express : Worker Threads pour le calcul, Crypto pour la signature, Web Streams API pour l'exposition.

## Objectifs

- Déporter un calcul CPU intensif dans un Worker Thread pour ne pas bloquer l'event loop
- Signer le résultat avec `node:crypto` (Ed25519) et fournir la signature au client
- Exposer le résultat sous forme de flux via la Web Streams API, en interopérant avec le stream Node attendu par Express 4

## Prérequis techniques

```bash
node --version            # v20.x ou v22.x attendu
mkdir tp7-service && cd tp7-service
npm init -y
npm pkg set type=module
npm install express
mkdir src
```

## Architecture cible

```
                 +------------------------+
   GET /compute  |   Route Express        |
 --------------> |  (thread principal)    |
                 +-----------+------------+
                             |
                  délègue le calcul (postMessage)
                             v
                 +------------------------+
                 |   Worker Thread        |
                 |  calcul CPU lourd      |
                 +-----------+------------+
                             |
                  résultat renvoyé au principal
                             v
                 +------------------------+
                 |  Crypto (Ed25519)      |
                 |  signature du résultat |
                 +-----------+------------+
                             |
              ReadableStream (Web Streams API)
                  -> Readable.fromWeb -> res
                             v
                       Client HTTP
```

## Étapes

### Étape 1 — Le worker de calcul (durée ~25 min)

Objectif : isoler le calcul CPU dans un Worker Thread, communiquant avec le thread principal.

- Créez `src/worker.js` : il reçoit une entrée via `workerData`, réalise un calcul bloquant volontairement coûteux (par exemple le calcul itératif d'un Fibonacci élevé, ou un hachage répété), puis renvoie le résultat avec `parentPort.postMessage(...)`.
- Créez `src/compute.js` exportant une fonction `runComputation(input)` qui instancie un `Worker`, transmet `input` via `workerData`, et retourne une `Promise` résolue sur le message du worker (et rejetée sur `error`).

Commandes utiles :

```bash
node -e "import('./src/compute.js').then(m => m.runComputation(35).then(console.log))"
```

Point de contrôle : un appel direct à `runComputation` renvoie un résultat et le thread principal n'est pas bloqué pendant le calcul (vous pouvez lancer deux calculs en parallèle et observer qu'ils ne se sérialisent pas sur l'event loop).

### Étape 2 — Signature du résultat (durée ~25 min)

Objectif : garantir l'intégrité du résultat avec une signature Ed25519.

- Créez `src/signer.js` : générez une paire de clés Ed25519 au démarrage (`generateKeyPairSync("ed25519")`), exportez une fonction `sign(payload)` qui retourne la signature (en base64) et une fonction `getPublicKeyPem()` exposant la clé publique.
- Le payload à signer est la représentation canonique du résultat (par exemple `JSON.stringify({ input, result })`).

Commandes utiles :

```bash
node -e "import('./src/signer.js').then(m => { const s = m.sign('test'); console.log(s); })"
```

Point de contrôle : la signature est reproductible pour un même payload et une même paire de clés, et la vérification (`verify`) renvoie `true` pour un payload intègre, `false` après altération.

### Étape 3 — Route Express en flux (durée ~40 min)

Objectif : assembler le tout dans une route `GET /compute` qui répond en flux.

- Créez `src/server.js` avec une route `GET /compute?input=...`.
- La route appelle `runComputation`, signe le résultat, construit un objet `{ input, result, signature, publicKey }`.
- Construisez un `ReadableStream` (Web Streams API) qui émet ce résultat (idéalement en plusieurs morceaux pour illustrer le streaming), puis convertissez-le en stream Node via `Readable.fromWeb(...)` et pipez-le dans `res`.
- Positionnez l'en-tête `Content-Type: application/json` et gérez les erreurs (calcul échoué, input invalide) avec un code HTTP adapté.

Commandes utiles :

```bash
node src/server.js
curl "http://localhost:3000/compute?input=35"
```

Point de contrôle : la réponse contient `input`, `result`, `signature` et `publicKey` ; un second client vérifiant la signature avec la clé publique retournée obtient un résultat valide ; le serveur reste réactif sur une autre route pendant un calcul long.

## Livrable attendu

Un dépôt `tp7-service` contenant `src/worker.js`, `src/compute.js`, `src/signer.js` et `src/server.js`, démarrable par `node src/server.js`, exposant `GET /compute` qui renvoie en flux un résultat de calcul signé et vérifiable.

## Dépannage

<details>
<summary>Le worker ne démarre pas / chemin introuvable</summary>

En modules ES, donnez au `Worker` une URL absolue : `new Worker(new URL("./worker.js", import.meta.url))`. Un chemin relatif simple peut échouer selon le répertoire de lancement.

</details>

<details>
<summary>"Cannot use import statement" dans le worker</summary>

Vérifiez que `package.json` contient `"type": "module"`. Les fichiers `.js` sont alors traités comme des modules ES, worker compris.

</details>

<details>
<summary>La réponse arrive vide ou tronquée</summary>

Assurez-vous de fermer le `ReadableStream` (`controller.close()`) après avoir émis tous les morceaux, et de ne pas appeler `res.end()` manuellement quand vous pipez : `pipe` ferme la réponse pour vous.

</details>

<details>
<summary>La signature ne se vérifie pas côté client</summary>

La vérification doit porter sur exactement le même payload que la signature (mêmes octets, même sérialisation). Signez et vérifiez la même chaîne canonique, sans réordonner les clés JSON entre les deux étapes.

</details>

## Pour aller plus loin

- Mettez en place un pool de workers (réutilisation plutôt qu'un worker par requête) pour réduire le coût de création de thread.
- Streamez réellement un gros résultat (génération progressive) pour mettre en évidence la backpressure.
- Exposez un endpoint `GET /public-key` et écrivez un petit client Node qui appelle `/compute` puis vérifie la signature de bout en bout.
- Comparez le comportement sous charge avec et sans Worker Thread (calcul directement dans la route).
