# TP 1 — Construire une API Express structurée et sécurisée

> Module 1 · Durée : 1h30 · Difficulté : 3/5 · Type : Travaux pratiques guidés

## Mise en situation

Utopios vous confie le démarrage d'une API de gestion de tâches (`tasks`). Le code prototype actuel entasse tout dans un seul fichier, ne gère pas les erreurs proprement et n'a aucune authentification. Votre mission : poser une base saine et reproductible, organisée en couches (routes, controllers, services), avec un middleware d'erreur centralisé, un guard d'authentification JWT et une validation des entrées. Les versions des dépendances seront épinglées pour garantir la reproductibilité d'une machine à l'autre.

## Objectifs

- Structurer une API Express en couches routes / controllers / services
- Mettre en place un middleware d'erreur centralisé compatible avec les handlers `async`
- Protéger des routes avec un guard d'authentification JWT
- Valider les données entrantes avant traitement
- Épingler les versions des dépendances dans `package.json`

## Prérequis techniques

Logiciels :

- Node.js 20 ou 22 LTS
- npm 10+
- Un client HTTP (curl, HTTPie ou une extension type REST client)

### Vérification de l'environnement

```bash
node --version   # doit afficher v20.x ou v22.x
npm --version    # doit afficher 10.x ou plus
mkdir -p api-tasks && cd api-tasks
npm init -y
```

## Architecture cible

```text
api-tasks/
├── package.json
├── .env
└── src/
    ├── server.js              point d'entrée, démarrage du serveur
    ├── app.js                 configuration Express et branchement
    ├── routes/
    │   ├── auth.routes.js      POST /auth/login
    │   └── tasks.routes.js     CRUD protégé /tasks
    ├── controllers/

    │   └── tasks.controller.js
    ├── services/
    │   ├── auth.service.js     génération/vérif JWT

```

## Étapes

### Étape 1 — Initialiser le projet avec versions épinglées (20 min)

Objectif : disposer d'un `package.json` reproductible et de la structure de dossiers.

Installez les dépendances en épinglant les versions exactes :

```bash
npm install --save-exact express@4.21.2 jsonwebtoken@9.0.2 zod@3.24.1 dotenv@16.4.7
```

Éditez `package.json` pour vérifier que les versions sont fixes (sans `^` ni `~`) et ajoutez le mode module et un script de démarrage :

```json
{
  "type": "module",
  "scripts": {
    "start": "node src/server.js"
  }
}
```

Créez la structure de dossiers et le fichier `.env` :

```bash
mkdir -p src/routes src/controllers src/services src/middlewares
printf 'JWT_SECRET=changez-moi-en-production\nPORT=3000\n' > .env
```

Point de contrôle : `cat package.json` montre des versions exactes pour `express`, `jsonwebtoken`, `zod` et `dotenv`, et les dossiers `src/...` existent.

### Étape 2 — Couches Express et middleware d'erreur (35 min)

Objectif : un serveur Express qui démarre, organisé en couches, avec gestion d'erreur centralisée.

Créez le service métier `src/services/tasks.service.js` (stockage en mémoire pour ce TP) avec les opérations `lister`, `creer` et `recupererParId`. Une tâche introuvable doit lever une erreur portant une propriété `status = 404`.

Créez le controller `src/controllers/tasks.controller.js` : il lit `req`, appelle le service, renvoie la réponse JSON et propage toute erreur via `next(err)`.

Créez le routeur `src/routes/tasks.routes.js` qui branche `GET /` et `POST /` sur le controller.

Créez le middleware d'erreur `src/middlewares/erreurs.js` (4 arguments) qui renvoie `{ error: ... }` avec le bon code HTTP.

Assemblez le tout dans `src/app.js` (configuration Express, `express.json()`, branchement des routes, middleware d'erreur en dernier) puis démarrez via `src/server.js`.

Lancez et testez :

```bash
npm start
# dans un autre terminal
curl http://localhost:3000/tasks
curl -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" \
  -d '{"titre":"Préparer la démo"}'
```

Point de contrôle : `GET /tasks` renvoie un tableau JSON, `POST /tasks` crée une tâche, et une erreur métier (id inexistant) renvoie un code `404` formaté par le middleware d'erreur, pas une stack trace brute.

### Étape 3 — Authentification JWT, guard et validation (35 min)

Objectif : protéger les routes d'écriture avec un token JWT et valider les entrées.

Créez `src/services/auth.service.js` qui génère un JWT (`jwt.sign`) pour un utilisateur de test, et le controller/route `POST /auth/login` qui renvoie ce token.

Créez le guard `src/middlewares/guard.js` : il lit l'en-tête `Authorization: Bearer <token>`, vérifie le token avec `jwt.verify`, attache `req.user` et appelle `next()`. En cas d'absence ou de token invalide, il répond `401`.

Créez `src/middlewares/validate.js` : un middleware générique qui valide `req.body` contre un schéma `zod` et renvoie `400` avec les détails en cas d'échec.

Protégez `POST /tasks` avec le guard et le middleware de validation.

Testez le parcours complet :

```bash
# 1. Sans token : refusé
curl -i -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" -d '{"titre":"X"}'

# 2. Obtenir un token
curl -s -X POST http://localhost:3000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"login":"demo","password":"demo"}'

# 3. Avec token (remplacez <TOKEN>)
curl -i -X POST http://localhost:3000/tasks \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"titre":"Tâche authentifiée"}'
```

Point de contrôle : sans token la création renvoie `401`, avec un token valide elle renvoie `201`, et un body invalide (titre manquant) renvoie `400` avec un message de validation.

## Dépannage courant

<details>
<summary>SyntaxError: Cannot use import statement outside a module</summary>

Le projet utilise la syntaxe `import`. Vérifiez que `package.json` contient `"type": "module"`. Sinon, renommez vos fichiers en `.mjs` ou ajoutez ce champ.

</details>

<details>
<summary>Le middleware d'erreur ne se déclenche pas sur une route async</summary>

Une erreur levée dans un handler `async` n'est pas capturée automatiquement par Express 4. Propagez-la avec `next(err)` dans un `try`/`catch`, ou enveloppez le handler dans un petit wrapper `(fn) => (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next)`.

</details>

<details>
<summary>JsonWebTokenError: invalid signature</summary>

Le `JWT_SECRET` utilisé pour signer diffère de celui utilisé pour vérifier. Assurez-vous que `dotenv` est chargé avant tout accès à `process.env.JWT_SECRET` et que le même secret sert dans `sign` et `verify`.

</details>

<details>
<summary>req.body est undefined</summary>

Le middleware `express.json()` n'est pas branché, ou l'en-tête `Content-Type: application/json` est absent de la requête. Vérifiez les deux.

</details>
