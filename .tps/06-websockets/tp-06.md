# TP 6 — Application de chat temps réel scalable avec Socket.IO

> Module 6 · Durée : ~1h30 · Difficulté : 4/5 · Type : Travaux pratiques guidés

## Mise en situation

Utopios vous confie la construction du service de messagerie temps réel d'une plateforme collaborative. Le service doit gérer plusieurs espaces (un namespace par produit), des salons dynamiques (rooms) à l'intérieur d'un espace, et n'accepter que des utilisateurs authentifiés. Surtout, il doit tenir la charge : l'application tourne en production sur plusieurs instances Node derrière un load balancer, et un message envoyé depuis une instance doit atteindre tous les destinataires, quelle que soit l'instance à laquelle ils sont connectés.

Vous allez construire ce service pas à pas, puis le lancer en deux instances partageant un Redis pour valider le scaling horizontal.

## Objectifs

- Structurer un serveur Socket.IO sur Express avec un namespace dédié au chat
- Authentifier le handshake par token avec le middleware `io.use()`
- Implémenter des rooms et des événements typés (message, présence) avec acknowledgements
- Mettre en place le scaling multi-instances avec `@socket.io/redis-adapter` et valider le routage inter-instances
- Orchestrer Redis via docker-compose

## Prérequis techniques

## Architecture cible

```
                    Navigateurs (clients)
                            |
                    Load Balancer / round-robin
                   /                          \
        Instance A (PORT 3000)        Instance B (PORT 3001)
        Express + Socket.IO           Express + Socket.IO
        namespace /chat               namespace /chat
        io.use(authToken)             io.use(authToken)
                   \                          /
                    @socket.io/redis-adapter
                            |
                    Redis (Pub/Sub) -- docker-compose
```

## Étapes

### Étape 1 — Serveur de base, namespace et authentification (durée : ~30 min)

Objectif : démarrer un serveur Express + Socket.IO avec un namespace `/chat` protégé par token.

1. Initialisez le projet dans `code/module-08/tp/` (ou copiez le squelette fourni) :

```bash
cd code/module-08/tp
npm init -y
npm pkg set type=module
npm install express socket.io socket.io-client jsonwebtoken
npm install @socket.io/redis-adapter redis
```

2. Créez `server.js` qui démarre Express, attache Socket.IO au serveur HTTP, et expose un namespace `/chat`.
3. Ajoutez un middleware `io.of("/chat").use((socket, next) => ...)` qui lit `socket.handshake.auth.token`, le vérifie avec `jsonwebtoken` et stocke l'utilisateur dans `socket.data.user`. En cas d'échec, appelez `next(new Error("unauthorized"))`.
4. Exposez une route Express `GET /token?user=alice` qui renvoie un JWT de test signé avec le même secret (uniquement pour le TP, pas en production).

Point de contrôle : une connexion sans token au namespace `/chat` déclenche un `connect_error`. Une connexion avec un token valide obtenu via `/token` réussit et logue le nom d'utilisateur côté serveur.

### Étape 2 — Rooms, événements typés et présence (durée : ~35 min)

Objectif : permettre de rejoindre des rooms et d'échanger des messages confirmés.

1. Dans le namespace `/chat`, gérez l'événement `joinRoom` (`{ room }`) : faites rejoindre la room, notifiez les autres membres via `presence` (`{ user, room, type: "join" }`), et renvoyez via l'ack la liste des membres actuels.
2. Gérez `leaveRoom` symétriquement avec un `presence` de type `"leave"`.
3. Gérez `chatMessage` (`{ room, text }` + ack) : refusez le message si l'émetteur n'est pas dans la room (`socket.rooms.has(room)`), sinon diffusez `chatMessage` (`{ user, room, text, at }`) à `io.of("/chat").to(room)` et confirmez via l'ack `{ status: "ok", at }`.
4. Créez un client de test : soit la page `public/index.html` servie par Express, soit un script `client.js` utilisant `socket.io-client`, qui récupère un token via `/token`, se connecte à `/chat`, rejoint une room et envoie des messages.

Point de contrôle : deux clients dans la room `general` voient leurs messages mutuels et reçoivent les `presence`. Un client dans la room `random` ne reçoit rien de `general`. L'émetteur reçoit bien l'ack `status: "ok"`.

### Étape 3 — Scaling multi-instances avec l'adaptateur Redis (durée : ~25 min)

Objectif : router les diffusions entre deux instances via Redis.

1. Créez `docker-compose.yml` lançant un service `redis` (image `redis:7-alpine`) exposé sur le port 6379.

```bash
docker compose up -d
docker compose ps
```

2. Dans `server.js`, après la création de `io`, branchez l'adaptateur Redis :

```js
import { createClient } from "redis";
import { createAdapter } from "@socket.io/redis-adapter";

const pubClient = createClient({ url: process.env.REDIS_URL });
const subClient = pubClient.duplicate();
await Promise.all([pubClient.connect(), subClient.connect()]);
io.adapter(createAdapter(pubClient, subClient));
```

3. Rendez le port configurable via `process.env.PORT` (défaut 3000) et lancez deux instances :

```bash
REDIS_URL=redis://localhost:6379 PORT=3000 node server.js
REDIS_URL=redis://localhost:6379 PORT=3001 node server.js
```

4. Connectez un client à l'instance du port 3000 et un autre à celle du port 3001, faites-les rejoindre la même room `general`.

Point de contrôle : un message envoyé par le client connecté au port 3000 est reçu par le client connecté au port 3001. En coupant Redis (`docker compose stop redis`), la diffusion inter-instances cesse, ce qui confirme le rôle de l'adaptateur.

## Dépannage courant

<details>
<summary>connect_error: unauthorized à la connexion</summary>

Le token n'est pas transmis ou est invalide. Vérifiez que le client passe `{ auth: { token } }` à `io(url + "/chat", ...)` et que le secret de signature (`/token`) est identique à celui de vérification dans `io.use`. Le namespace doit être présent dans l'URL de connexion.

</details>

<details>
<summary>Les messages ne passent pas entre les deux instances</summary>

L'adaptateur Redis n'est pas actif ou les deux instances ne pointent pas vers le même Redis. Vérifiez que `docker compose ps` montre Redis en cours d'exécution et que `REDIS_URL` est identique pour les deux processus. Confirmez que `io.adapter(createAdapter(...))` est bien appelé avant `listen`.

</details>

<details>
<summary>Error: Cannot use import statement outside a module</summary>

Le projet n'est pas en mode ES. Exécutez `npm pkg set type=module` ou ajoutez `"type": "module"` au `package.json`.</details>

<details>
<summary>CORS bloque le client navigateur</summary>

Configurez les options CORS de Socket.IO : `new Server(httpServer, { cors: { origin: "http://localhost:3000" } })`, ou servez le client depuis Express pour rester en même origine.

</details>

## Pour aller plus loin

- Persistez l'historique des derniers messages d'une room dans Redis et renvoyez-le au moment du `joinRoom`.
- Ajoutez un indicateur "en train d'écrire" (`typing`) diffusé en temps réel aux membres de la room.
- Mettez un rate limit par socket sur l'événement `chatMessage` pour résister au flood.
- Placez un vrai load balancer (nginx) devant les deux instances avec WebSocket et observez le comportement.
