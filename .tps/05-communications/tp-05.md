# TP 5 — Découpler une API Express d'un worker via AMQP

> Module 5 / Gestion avancée des flux Node.js — TP guidés — Difficulté 4/5 — Durée ~1h30

## Mise en situation

Votre API Express expose un endpoint de création de rapports PDF. La génération est coûteuse (plusieurs secondes) et bloque actuellement la réponse HTTP. Sous charge, l'API s'effondre. Vous allez la découpler : l'API publiera une tâche dans une file RabbitMQ et répondra immédiatement, tandis qu'un worker indépendant consommera les tâches une par une, gérera les erreurs et utilisera le `prefetch` pour réguler la back-pressure.

## Objectifs

- Publier une tâche depuis Express sur un exchange AMQP et répondre `202 Accepted`
- Consommer les tâches dans un worker avec acquittement manuel
- Gérer les erreurs : requeue pour les erreurs transitoires, dead-letter pour les erreurs définitives
- Maîtriser la back-pressure du worker avec `prefetch`

## Prérequis techniques

- Node.js 20 ou 22 installé
- Docker installé et démarré (pour RabbitMQ et Redis)
- Un terminal et un éditeur de code

```bash
# Démarrer RabbitMQ avec l'interface de gestion (port 15672)
docker run -d --name rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management

# Démarrer Redis (utilisé en bonus pour la diffusion de progression)
docker run -d --name redis -p 6379:6379 redis:7

# Vérifier que les conteneurs tournent
docker ps

# Initialiser le projet
mkdir tp-amqp && cd tp-amqp
npm init -y
npm install express amqplib
```

L'interface RabbitMQ est accessible sur http://localhost:15672 (login : `guest` / `guest`).

## Architecture cible

```
  +-------------+   POST /rapports    +------------------+
  |  Client     | ------------------> |  API Express     |
  +-------------+   202 Accepted      |  (producteur)    |
                                      +--------+---------+
                                               | publish
                                               v
                                  exchange "rapports" (direct)
                                               | routing key
                                               | "rapport.demande"
                                               v
                                      +------------------+
                                      | queue            |
                                      | "generation"     |
                                      | prefetch = 2     |
                                      +--------+---------+
                                          ack  | nack(false)
                                  +------------+------------+
                                  |                         |
                                  v                         v
                          +--------------+        exchange "dlx"
                          | Worker       |             |
                          | (consommateur)|            v
                          +--------------+     queue "generation.dlq"
```

## Étape 1 — Mettre en place la topologie et le producteur

**Durée :** ~30 min
**Objectif :** créer l'exchange, la queue durable avec sa DLQ, et un endpoint Express qui publie une tâche.

Créez un module `topologie.js` qui déclare la topologie (idempotent grâce à `assert`), puis `api.js` qui se connecte, publie et répond `202`.

```bash
# Lancer l'API
node api.js

# Dans un autre terminal, envoyer une demande
curl -X POST http://localhost:3000/rapports \
  -H "Content-Type: application/json" \
  -d '{"titre":"Bilan Q1","client":"Utopios"}'
```

La réponse doit être immédiate, du type `{ "statut": "accepte", "id": "..." }`.

**Point de contrôle :** dans l'interface RabbitMQ (onglet Queues), la queue `generation` affiche un message en attente (« Ready ») après chaque appel `curl`.

## Étape 2 — Implémenter le worker avec ack et prefetch

**Durée :** ~30 min
**Objectif :** consommer les tâches une par une, simuler un traitement, acquitter après succès et observer la régulation par `prefetch`.

Créez `worker.js` qui réutilise `topologie.js`, fixe `ch.prefetch(2)` et consomme. Simulez un traitement de 3 secondes avec un `await new Promise(r => setTimeout(r, 3000))` avant le `ack`.

```bash
# Lancer le worker
node worker.js

# Envoyer plusieurs demandes d'affilée et observer
curl -X POST http://localhost:3000/rapports -H "Content-Type: application/json" -d '{"titre":"R1"}'
curl -X POST http://localhost:3000/rapports -H "Content-Type: application/json" -d '{"titre":"R2"}'
curl -X POST http://localhost:3000/rapports -H "Content-Type: application/json" -d '{"titre":"R3"}'
```

**Point de contrôle :** dans l'interface RabbitMQ, la colonne « Unacked » ne dépasse jamais 2 (valeur du prefetch), même avec 3 messages envoyés. Les messages sont traités par lots de 2.

## Étape 3 — Gérer les erreurs : requeue et dead-letter

**Durée :** ~30 min
**Objectif :** distinguer erreur transitoire et erreur définitive, et router les échecs définitifs vers la DLQ.

Dans le worker, ajoutez une logique : si le titre contient `KO_DEFINITIF`, faites un `ch.nack(msg, false, false)` (vers DLQ) ; si le titre contient `KO_RETRY`, faites un `ch.nack(msg, false, true)` (requeue), en limitant le nombre de tentatives via l'en-tête `x-death`.

```bash
# Erreur définitive -> doit atterrir en DLQ
curl -X POST http://localhost:3000/rapports -H "Content-Type: application/json" -d '{"titre":"KO_DEFINITIF"}'

# Erreur transitoire -> requeue
curl -X POST http://localhost:3000/rapports -H "Content-Type: application/json" -d '{"titre":"KO_RETRY"}'
```

**Point de contrôle :** après l'envoi de `KO_DEFINITIF`, la queue `generation.dlq` contient un message dans l'interface RabbitMQ, et la queue `generation` est revenue à zéro message non acquitté.

## Livrable attendu

- `topologie.js` : déclaration idempotente de l'exchange `rapports`, de la queue `generation` (durable, avec arguments de dead-letter), de l'exchange `dlx` et de la queue `generation.dlq`
- `api.js` : API Express qui publie en `persistent` et répond `202`
- `worker.js` : consommateur avec `prefetch(2)`, `ack` après succès, `nack` différencié selon le type d'erreur
- Capture (ou description) de l'interface RabbitMQ montrant le bon comportement aux trois points de contrôle

## Dépannage

<details>
<summary>« ECONNREFUSED 127.0.0.1:5672 »</summary>

RabbitMQ n'est pas démarré ou pas encore prêt. Vérifiez `docker ps`. Le conteneur met quelques secondes à accepter les connexions après le `docker run` ; relancez votre script.
</details>

<details>
<summary>Les messages restent en « Unacked » et ne disparaissent jamais</summary>

Vous avez oublié le `ch.ack(msg)` après le traitement, ou une exception a court-circuité l'appel. Vérifiez que le `ack`/`nack` est appelé dans tous les chemins (try/catch).
</details>

<details>
<summary>La DLQ reste vide après un nack</summary>

Vérifiez que la queue `generation` a bien été créée avec l'argument `x-dead-letter-exchange` pointant vers `dlx`, et que `dlx` est bindé à `generation.dlq`. Si la queue existait déjà sans ces arguments, supprimez-la dans l'interface puis relancez (RabbitMQ refuse de modifier les arguments d'une queue existante).
</details>

<details>
<summary>Le prefetch ne semble pas avoir d'effet</summary>

`ch.prefetch()` doit être appelé sur le channel **avant** `ch.consume()`. Vérifiez aussi que vous êtes bien en acquittement manuel (`{ noAck: false }`, qui est la valeur par défaut).
</details>

## Pour aller plus loin

- Lancez deux instances du worker (`node worker.js` dans deux terminaux) et observez la répartition des tâches entre elles (round-robin équitable grâce au prefetch).
- Ajoutez une queue de retry avec TTL pour temporiser les requeue au lieu de boucler immédiatement.
- Bonus Redis : publiez la progression de chaque rapport sur un canal Redis et affichez-la côté API via un endpoint SSE, pour diffuser l'avancement à plusieurs instances.
